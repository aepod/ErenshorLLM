# ErenshorLLMDialog

**LLM-Powered NPC Dialog for Erenshor** | [GitHub](https://github.com/aepod/ErenshorLLM) | [Releases](https://github.com/aepod/ErenshorLLM/releases) | [Issues](https://github.com/aepod/ErenshorLLM/issues)

Replaces SimPlayer canned chat with dynamic, personality-driven responses generated by a local language model. Each of the 160 named SimPlayers has a unique personality profile -- archetypes, speech patterns, quirks, and vocabulary -- grounded in Erenshor world lore. SimPlayers you meet without a personality file get one generated on-the-fly from their in-game traits, so every NPC sounds distinct.

Works entirely offline. No cloud account needed.

## Requirements

- [Erenshor](https://store.steampowered.com/app/2382520/Erenshor/) (Steam)
- [BepInEx 5.4.x](https://github.com/BepInEx/BepInEx/releases) installed in your Erenshor game directory
- Windows x64
- ~1.5 GB disk space (mod + models)
- GPU recommended (NVIDIA CUDA or Vulkan) but CPU inference works

## Installation

1. Install BepInEx 5 into your Erenshor directory if you haven't already.
2. Extract the mod zip into your BepInEx plugins folder:

```
Erenshor/
  BepInEx/
    plugins/
      ErenshorLLMDialog/           <-- extract here
        ErenshorLLMDialog.dll      # BepInEx plugin
        erenshor-llm.exe           # Intelligence sidecar
        shimmy.exe                 # Local LLM inference server
        onnxruntime.dll            # ONNX embedding runtime
        data/
          erenshor-llm.toml        # Sidecar config
          grounding.json           # Entity validation data
          dist/                    # Pre-built vector indexes
            lore.ruvector
            responses.ruvector
            memory.ruvector
            personality.ruvector
          models/                  # AI models
            all-minilm-l6-v2.onnx # Embedding model (~86 MB)
            tokenizer.json
          personalities/           # 160 SimPlayer personality files
          templates/               # Response template library
```

3. **Download a GGUF model** (see [Model Setup](#model-setup) below).
4. Launch Erenshor. The mod auto-starts the sidecar and shimmy processes.

## Model Setup

The mod needs a GGUF language model for text generation. Place it in the `data/models/` directory.

**Recommended: Qwen 2.5 0.5B Instruct (Q4_K_M)**

Small, fast, runs on any hardware. ~470 MB download. Others were tried during development, a smaller model would be nice but loses coherency. Long term we may publish a trained model for this, especially the paraphrase pipeline.

Download from HuggingFace:
- [Qwen2.5-0.5B-Instruct-GGUF on HuggingFace](https://huggingface.co/Qwen/Qwen2.5-0.5B-Instruct-GGUF)
- Download the `qwen2.5-0.5b-instruct-q4_k_m.gguf` file
- Place in `ErenshorLLMDialog/data/models/`

### Cloud Mode (OpenRouter)

Instead of running a local model, you can use a free cloud model through [OpenRouter](https://openrouter.ai/). Set `LLM Mode` to `Cloud` or `Hybrid` in BepInEx config and add your OpenRouter API key to `data/erenshor-llm.toml`.

**Best free model: [`liquid/lfm-2.5-1.2b-instruct:free`](https://openrouter.ai/liquid/lfm-2.5-1.2b-instruct:free)**

After benchmarking all [free models on OpenRouter](https://openrouter.ai/models?max_price=0), `liquid/lfm-2.5-1.2b-instruct:free` scored highest across latency, response quality, entity preservation, and output length. It produces concise, in-character responses without slop and handles Erenshor entity names well.

To configure cloud mode, edit `data/erenshor-llm.toml`:
```toml
[llm]
enabled = true
mode = "cloud"

[llm.cloud]
provider = "openrouter"
api_key = "sk-or-v1-your-key-here"
api_endpoint = "https://openrouter.ai/api/v1"
model = "liquid/lfm-2.5-1.2b-instruct:free"
timeout_ms = 10000
```

Get a free API key at [openrouter.ai](https://openrouter.ai). Free tier models have rate limits but work well for Erenshor's dialog pacing.

## How It Works

The mod has two parts that communicate over HTTP on your machine:

```
Erenshor (Unity)                     erenshor-llm (Rust sidecar)
+---------------------------+        +---------------------------+
|  ErenshorLLMDialog.dll    |  HTTP  |  Intelligence Engine      |
|  - Intercepts chat events |------->|  - Template matching      |
|  - Sends context to       |  :11435|  - Lore retrieval (RAG)   |
|    sidecar via HTTP       |<-------|  - Personality selection   |
|  - Injects LLM response   |        |  - LLM text generation    |
|    back into game chat    |        |    (via shimmy on :8012)  |
+---------------------------+        +---------------------------+
```

The **C# plugin** (ErenshorLLMDialog.dll) hooks into Erenshor's chat system using [Harmony](https://harmony.pardeike.net/) patches. When you type a message, the plugin captures it along with everything the game knows -- your zone, chat channel, which SimPlayers are nearby, your relationship level with the target sim, your guild, your class and level. It sends all of this to the sidecar as a single HTTP request and waits for a response to inject back into the game's chat window.

The **Rust sidecar** (erenshor-llm.exe) is where the intelligence lives. It runs a multi-stage pipeline that combines several AI techniques to produce responses that feel natural and in-character. Here's what happens for every chat message:

### The Response Pipeline

**Step 1 -- Embedding**

Your chat message is converted into a numerical representation called an [**embedding**](https://en.wikipedia.org/wiki/Word_embedding) -- a list of 384 numbers that capture the meaning of what you said. This is done by a small neural network called [**all-MiniLM-L6-v2**](https://huggingface.co/sentence-transformers/all-MiniLM-L6-v2) running locally through [ONNX Runtime](https://onnxruntime.ai/). The key insight is that similar messages produce similar embeddings: "hey want to group up?" and "looking for a party" end up as nearly identical number sequences, even though the words are completely different.

**Step 2 -- SONA Enhancement**

Before searching, the embedding is optionally refined by **SONA** (Self-Optimizing Neural Architecture). SONA is an adaptive learning system that adjusts search queries based on what has worked well in the past. It uses a technique called **MicroLoRA** -- a very small [LoRA](https://arxiv.org/abs/2106.09685) transformation (rank 1, just a few hundred parameters) that nudges the embedding toward templates and lore passages that previously led to good responses.

Think of it like this: if the system notices that combat-related questions consistently get the best results when matched with a particular cluster of templates, SONA subtly shifts future combat queries closer to that cluster. It learns continuously during play without any manual tuning.

**Step 3 -- Template Search (Vector Database)**

The enhanced embedding is used to search a [**vector database**](https://en.wikipedia.org/wiki/Vector_database) -- a specialized database that finds entries by meaning rather than exact text matching. The mod ships with ~660 hand-written response templates covering greetings, combat talk, lore discussion, zone-specific banter, farewell messages, and more. Each template was pre-embedded during the build process, and searching finds the top 10 candidates whose meanings are closest to what you said.

The vector database uses an algorithm called [**HNSW**](https://arxiv.org/abs/1603.09320) (Hierarchical Navigable Small Worlds) stored in **ruvector** format. HNSW builds a graph of connections between entries so that searching millions of vectors takes milliseconds instead of scanning every single one.

**Step 4 -- Re-Ranking**

The raw search results are good but generic -- they only consider what your message means, not who's saying it or where you are. The **re-ranker** scores each candidate on five weighted factors:

- **Semantic similarity** (20%) -- How well the template matches what you said
- **Personality fit** (30%) -- Whether the template matches the SimPlayer's traits (social, friendly, scholarly, aggressive). A warrior-type sim ranks combat templates higher; a lore keeper ranks knowledge templates higher
- **Zone affinity** (20%) -- Whether the template is tagged for the current zone. A template about "the ancient ruins here" only scores well in zones with ruins
- **Channel match** (15%) -- Whether the template fits the chat channel. A guild recruitment message won't fire in whisper
- **Relationship level** (15%) -- Whether the template's tone matches how the sim feels about you. A sim who likes you uses friendlier templates

The re-ranker also applies **recency tracking** -- if a sim just used a particular template, it gets penalized so the same character doesn't repeat themselves. Different sims can still use the same template independently.

**Step 5 -- Lore and Memory Search**

In parallel with template search, the sidecar searches two more vector databases:

- **Lore index** (~2,900 passages) -- Curated world knowledge from the [Erenshor wiki](https://erenshor.wiki.gg/): zone descriptions, enemy lore, item histories, NPC backstories, quest information. This is what makes a sim able to say "The Shimmerfen is known for its rare Moonpetal flowers" instead of making something up. This technique is called [**RAG**](https://en.wikipedia.org/wiki/Retrieval-augmented_generation) (Retrieval-Augmented Generation) -- instead of relying on the LLM to know everything, we retrieve relevant facts and feed them into the prompt so the model works with real information.

- **Memory index** -- Conversation history. When you talk to a sim about hunting in Stormwall, that interaction is stored. Next time the topic comes up, the memory is retrieved and the sim can reference it: "Last time you mentioned the drakes in Stormwall..."

**Step 6 -- Template Enrichment**

The winning template's placeholder tokens are expanded with real game data:

- `{player}` becomes your character name
- `{sim}` becomes the SimPlayer's name
- `{zone}` becomes the current zone
- `{mob}` and `{item}` are filled from lore search results when available

This turns a generic template like "Hey {player}, have you fought any {mob} in {zone}?" into "Hey Valorian, have you fought any Thornback Spiders in Shimmerfen?"

**Step 7 -- LLM Decision (Three Paths)**

Now the system decides how to produce the final response. There are three possible paths:

- **Path A -- Full LLM Generation:** If the best template scored below the confidence threshold (default 0.70), the template isn't a great match. The [**LLM**](https://en.wikipedia.org/wiki/Large_language_model) (Large Language Model) generates a completely original response using the SimPlayer's personality profile, retrieved lore, conversation memory, and the game context as its prompt. This is the most creative path but also the slowest (~1-5 seconds depending on your hardware).

- **Path B -- Template Paraphrase:** If the template scored high but was recently used, the LLM rewrites the template text in the sim's voice to add variety. Same meaning, different words. This triggers 80% of the time for recently-used templates, 90% for sim-to-sim ambient chatter, and 15% randomly otherwise.

- **Path C -- Fast Path (Verbatim):** If the template scored high and hasn't been used recently, it's sent back as-is. This is instant (~50ms total pipeline time) and is what happens most often.

The LLM runs through [**shimmy**](https://github.com/Michael-A-Kuykendall/shimmy), a local inference server that loads a [**GGUF**](https://huggingface.co/docs/hub/gguf) model file (a compressed format for running language models efficiently). Shimmy auto-detects your GPU and uses [CUDA](https://developer.nvidia.com/cuda-toolkit) (NVIDIA), [Vulkan](https://www.vulkan.org/) (AMD/Intel), or falls back to CPU.

**Step 8 -- Post-Processing and GEPA Validation**

Raw LLM output goes through cleanup:

- Strips markdown formatting, emote markers, and instruction leakage (when the model accidentally echoes its own prompt)
- Collapses repetition (small models sometimes loop: "the sword the sword the sword" becomes "the sword")
- Truncates at a natural sentence boundary so responses don't end mid-thought
- **GEPA** (Game Entity Prompt Anchoring) validation checks that any zone, item, NPC, enemy, or class names in the response are real Erenshor entities. If the LLM hallucinated a zone called "Darkhollow" that doesn't exist, GEPA strips or replaces it. This is powered by `grounding.json`, a master list of every known game entity.

**Step 9 -- SONA Learning**

After a successful LLM response, SONA records a **trajectory** -- the original query embedding paired with the LLM's output embedding. Over time, these trajectories teach SONA which search directions lead to the best template matches, forming a feedback loop that improves response quality the more you play. Trajectories are processed in background cycles every 5 minutes.

**Step 10 -- Memory Storage**

The interaction is saved to the memory vector database so the sim can reference it in future conversations.

### Multi-Sim Responses

When you speak in say, guild, or shout channels, the plugin doesn't just ask one sim to respond. It identifies other eligible SimPlayers (nearby for say, guild members for guild, zone-wide for shout) and dispatches additional requests to the sidecar, each with a different sim name. Each sim runs the full pipeline independently with their own personality, producing a chorus of distinct responses with staggered timing.

SimPlayers can also react to each other's messages (**sim-to-sim chaining**). If Scrubby says something in response to you, Blademann might react to Scrubby's message. This is gated by a rate limiter and configurable depth limit to prevent runaway conversation chains.

### Chat Interception Architecture (v0.3.0+)

Starting with v0.3.0, the plugin uses a single unified `ChatInterceptHook` that replaces the previous three-hook system (`AddStringForDisplayPatch`, `LogAddColoredPatch`, `LogAddPlainPatch`). This was made possible by the March 10 game update which introduced `UpdateSocialLog.GlobalAddLine(ChatLogLine)` -- a single funnel point for all chat messages in the game.

The hook patches `GlobalAddLine` and uses the `ChatLogLine.LogType` flags for canonical channel detection, replacing the fragile color-string parsing from earlier versions. The classification pipeline runs in order:

1. **LogType mapping** -- `LogTypeMapper` converts game `LogType` flags to the mod's `ChatChannel` enum
2. **Message parsing** -- `MessageParser` extracts speaker name, raw text, and metadata from the `ChatLogLine`
3. **Known sim check** -- `IsKnownSim` determines if the speaker is a SimPlayer (vs. the real player or system)
4. **Data-only filter** -- `IsDataOnly` skips messages that are pure system/data lines (loot rolls, XP gains)
5. **Combat callout detection** -- `IsCombatCallout` identifies combat phrases for fast-path handling

Messages take one of two execution paths:
- **Sync combat template path** (<1ms) -- Combat callouts are matched instantly against `TemplateCache` and re-injected with personality variants. No async work, no sidecar round-trip.
- **Async dialog paraphrase path** -- All other messages are sent to the sidecar for full pipeline processing (embedding, search, LLM generation).

A re-entry guard prevents infinite loops when re-injected text flows back through `GlobalAddLine`. The hook includes graceful fallback to the legacy three-hook system on pre-March 10 game builds that lack the `GlobalAddLine` method. All exceptions are caught and logged -- the hook never propagates errors to the game engine.

### Combat Template System (v0.3.0+)

The `TemplateCache` provides instant variant lookups for 14 classified combat callout types:

| Type | Example Phrases |
|------|----------------|
| `pulling` | "Pulling!", "Inc!" |
| `aggro` | "I have aggro!", "It's on me!" |
| `oom` | "OOM!", "No mana!" |
| `healing` | "Healing you!", "Incoming heal!" |
| `heal_request` | "Need heal!", "Heal me!" |
| `retreat` | "Fall back!", "Run!" |
| `ready_check` | "Ready?", "Ready check!" |
| `taunting` | "Taunting!", "I'll grab it!" |
| `targeting` | "Targeting the add!", "Focus fire!" |
| `casting` | "Casting!", "Channeling!" |
| `cc_notice` | "CC'd!", "Mezzed!" |
| `stance` | "Going defensive!", "Switching stance!" |
| `close_call` | "That was close!", "Barely survived!" |
| `lfg` | "LFG!", "Looking for group!" |

Templates are selected based on personality match -- each variant is tagged with a **style** (nice, tryhard, mean) and **role** (tank, healer, dps). A tryhard tank who calls "Pulling!" gets something like "Incoming, burn it down!" while a nice healer gets "Pulling carefully, heals are ready!"

**Cache miss handling:** When a combat callout has no matching template variant, the cache queues a background LLM generation request via the sidecar's `/v1/templates/queue` endpoint. The original text is used as-is for the current message while the LLM generates new variants asynchronously. Generated templates persist to disk (`template_store.json`) via atomic write and survive across sessions.

**Personalization:** The `{speaker}` placeholder is expanded to the SimPlayer's name, and `PersonalizeString()` applies character quirks -- caps for characters who type in all caps, typos for clumsy typists, third-person references for characters like Scrubby.

**Zone self-reference filtering:** Templates used in party and guild channels avoid zone-specific language ("here in [place]") since group members may be in different zones. Templates used in shout and say channels allow zone references since all recipients are in the same zone.

**Graceful degradation:** If the sidecar is down or unreachable, the template cache serves whatever it has locally. If it has nothing, the original game text passes through unchanged with zero delay.

**Periodic refresh:** Every 60 seconds, the cache checks the sidecar for newly generated templates and merges them into its in-memory store.

### Template API Endpoints

The sidecar exposes four endpoints for the combat template system:

| Method | Endpoint | Description |
|--------|----------|-------------|
| `POST` | `/v1/templates/queue` | Queue background template generation for a trigger/style/role combination. Returns `202 Accepted` immediately. |
| `GET` | `/v1/templates/lookup` | Look up personality-matched template variants for a given trigger type, style, and role. Returns cached results. |
| `GET` | `/v1/templates/stats` | Template store statistics: total templates, variants per trigger, cache hit rate, generation queue depth. |
| `POST` | `/v1/templates/import` | Bulk import template packs (JSON array of template objects). Used for distributing community-created template sets. |

### Dynamic Personality Generation

The mod ships with 160 hand-crafted personality files, but the game can create SimPlayers that don't have one. When the sidecar encounters an unknown sim name, it generates a personality on-the-fly using:

1. **Game trait flags** sent by the C# plugin (friendly, aggressive, scholarly, social, grumpy, etc.)
2. A **deterministic hash** of the sim's name for consistent variation

The generated personality maps trait combinations to archetypes (aggressive + social = "raid leader", scholarly + friendly = "helpful sage"), picks appropriate tone, vocabulary, and speech patterns, then caches it for the rest of the session. The same unknown sim always gets the same generated personality.

### Field Reference

**Core identity** -- these are the most impactful fields to edit:

| Field | Type | What it does |
|-------|------|-------------|
| `name` | string | Must match the SimPlayer's in-game name (case-insensitive) |
| `archetype` | string | Short character concept. Used to derive trait flags (social, friendly, scholarly, aggressive) and included in the LLM system prompt |
| `tone` | string | How they speak. Goes directly into the LLM prompt |
| `vocabulary` | string[] | Words/phrases they favor. Included in the LLM prompt |
| `speech_patterns` | string[] | Descriptions of how they talk. Included in the LLM prompt |
| `knowledge_areas` | string[] | What topics they know about. Affects lore retrieval weighting |
| `quirks` | string[] | Behavioral/speech quirks. Included in the LLM prompt |
| `example_phrases` | string[] | Example dialog lines. Used for training data and as few-shot examples. More is better |

**Chat modifiers** -- control text formatting:

| Field | Type | Effect |
|-------|------|--------|
| `types_in_all_caps` | bool | ALL CAPS OUTPUT |
| `types_in_all_lowers` | bool | all lowercase output |
| `types_in_third_person` | bool | "Scrubby thinks this is great" instead of "I think this is great" |
| `loves_emojis` | bool | Sprinkles emojis into responses |
| `typo_rate` | float | 0.0 = no typos, 10.0+ = frequent typos |
| `abbreviates` | bool | Uses common chat abbreviations |
| `refers_to_self_as` | string/null | Custom self-reference (e.g. "this one", "the great Azzian") |
| `sign_off_lines` | string[] | Closing lines the character may use |

**Personality type** -- affects template matching:

| Value | Type | Description |
|-------|------|-------------|
| 1 | Nice | Friendly, helpful, encouraging |
| 2 | Tryhard | Competitive, min-maxing, intense |
| 3 | Mean | Rude, dismissive, aggressive |
| 5 | Neutral | Balanced, no strong lean |

**Behavioral attributes** -- influence response style:

| Field | Range | Description |
|-------|-------|-------------|
| `lore_chase` | 0-10 | How much they care about lore/knowledge |
| `gear_chase` | 0-10 | How much they talk about gear/items |
| `social_chase` | 0-10 | How social/talkative they are |
| `troublemaker` | 0-10 | Tendency to stir up trouble |
| `dedication_level` | 0-10 | How seriously they take the game |

**Special flags:**

| Field | Effect |
|-------|--------|
| `rival` | true = member of Friends' Club (rival faction). Gets special LLM prompt framing |
| `is_gm_character` | true = GM/developer character. Gets special treatment |

### Editing Tips

- **File name must match the `name` field** (lowercase). `scrubby.json` contains `"name": "Scrubby"`.
- **`_default.json`** is the fallback personality used when a SimPlayer has no dedicated file and no game traits. Edit this to change the baseline voice.
- You can **create new personality files** for SimPlayers that don't have one. Just create `<name>.json` with the character's in-game name.
- The `archetype` and `knowledge_areas` fields drive trait derivation (social, friendly, scholarly, aggressive). Keywords like "warrior", "battle", "combat" trigger the aggressive trait; "lore", "sage", "wisdom" trigger scholarly; "guild", "party", "team" trigger social; "helpful", "warm", "cheerful" trigger friendly.
- `example_phrases` are the single best way to influence how a character sounds. Add more examples to get more consistent output from the LLM.

### Applying Personality Changes

After editing personality files, the changes take effect on the next sidecar restart. There are two ways to do this:

**Option A: In-game restart (easiest)**

Toggle `Restart Sidecar` to `On` in the BepInEx config (via ConfigManager or by editing the `.cfg` file). The sidecar detects that personality files changed (via hash comparison) and automatically rebuilds the personality vector index before accepting requests.

**Option B: Manual rebuild**

If the sidecar is not running, you can rebuild the personality index directly:

```
erenshor-llm.exe --data-dir data build-personalities
```

This reads all `.json` files in `data/personalities/`, generates embeddings, and writes `data/dist/personality.ruvector`. The next time the sidecar starts, it will use the updated index.

The `build-personalities` command is safe to run against your deployed `data/` directory -- it only reads from `data/personalities/` and writes to `data/dist/personality.ruvector`. Unlike the other build commands, it does not require source repo files.

## erenshor-llm.exe -- Sidecar Reference

The sidecar binary serves as both the runtime intelligence server and an offline data processing tool. When launched without a subcommand, it starts the HTTP server. Subcommands are used for data pipeline operations.

### Server Mode (Default)

```
erenshor-llm.exe [OPTIONS]
```

Starts the intelligence sidecar HTTP server. This is what the BepInEx plugin auto-launches.

| Option | Default | Description |
|--------|---------|-------------|
| `--port` | 11435 | HTTP server port |
| `--config` | (auto) | Path to TOML config file |
| `--data-dir` | `.` | Root data directory |
| `--threads` | (config) | ONNX embedding thread count |
| `--log-format` | pretty | Log format: `pretty` or `json` |

### User-Safe Commands

These commands can be run against your deployed mod `data/` directory.

#### build-personalities

Rebuild the personality vector database from JSON personality files.

```
erenshor-llm.exe --data-dir data build-personalities [--input personalities] [--output dist/personality.ruvector]
```

Run this after editing personality files in `data/personalities/`. The sidecar also auto-rebuilds at startup when it detects personality file changes, so you usually don't need to run this manually -- just use `Restart Sidecar` from BepInEx config.

See [Customizing Personalities](#customizing-personalities) for details on editing personality files.

### Developer-Only Commands

**WARNING:** The following commands are build-time tools used during mod development. They require source data files (lore markdown, wiki dumps, grounding cross-references) that are **not included in the mod download**. These files live in the mod's source repository. Running these commands against the deployed `data/` directory will either fail or produce incomplete results.

If you are a mod developer working from the source repo, run these from the `sidecar/` directory with `--data-dir data`.

#### build-index

Build the lore vector database from curated markdown files.

```
erenshor-llm.exe --data-dir data build-index [--input lore] [--output dist/lore.ruvector]
```

Reads `.md` files from the input directory, generates embeddings, and writes a `.ruvector` HNSW database. Requires the ONNX embedding model and source lore markdown files (not shipped with the mod).

#### build-responses

Build the response template vector database from JSON template files.

```
erenshor-llm.exe --data-dir data build-responses [--input templates] [--output dist/responses.ruvector]
```

Requires the source template JSON files. The deployed `data/templates/` directory does contain these, but the command also requires the ONNX embedding model to generate vectors.

#### init-data

Reset all vector databases and rebuild everything from source.

```
erenshor-llm.exe --data-dir data init-data
```

**Destructive:** Deletes all `.ruvector` files and `memory.json`, then rebuilds lore, response, and personality indexes from source. Requires all source data. **Do not run this on a deployed mod** unless you have the full source repo -- you will lose your pre-built indexes and the sidecar will not function.

#### import-wiki

Import Erenshor wiki dump files into the curated lore directory.

```
erenshor-llm.exe --data-dir data import-wiki --wiki-dir /path/to/wiki/dump
```

Reads `.md` files with YAML frontmatter, cleans wiki markup, and organizes by category (zones, enemies, items, npcs, etc.). Does not generate embeddings -- run `build-index` afterward. Requires a wiki dump (not shipped).

#### clean-items

Transform raw wiki item stat dumps into conversational prose.

```
erenshor-llm.exe --data-dir data clean-items
```

Cross-references enemies and zones directories for drop source grounding. Run after `import-wiki` and before `build-index`. Requires source lore directories.

#### export-training

Export training data for [LoRA](https://arxiv.org/abs/2106.09685) fine-tuning in [ChatML](https://github.com/openai/openai-python/blob/main/chatml.md), Alpaca, and/or [ShareGPT](https://huggingface.co/docs/trl/dataset_formats#sharegpt) formats.

```
erenshor-llm.exe --data-dir data export-training [OPTIONS]
```

| Option | Default | Description |
|--------|---------|-------------|
| `--output-dir` | dist/training | Output directory |
| `--format` | all | chatml, alpaca, sharegpt, or all |
| `--strategies` | all | phrases, crossover, lore, multiturn |
| `--seed` | 42 | Deterministic RNG seed |
| `--max-per-strategy` | 0 | Cap per strategy (0=unlimited) |
| `--personality-types` | (all) | Filter: 1=Nice, 2=Tryhard, 3=Mean, 5=Neutral |
| `--categories` | (all) | Filter template categories |
| `--zones` | (all) | Filter zones |

Generates 30,000-50,000+ training pairs from personality phrases, template crossovers, lore Q&A, and multi-turn conversations. Requires all source data.

#### fine-tune

Generate training scripts and optionally run fine-tuning.

```
erenshor-llm.exe --data-dir data fine-tune [OPTIONS]
```

| Option | Default | Description |
|--------|---------|-------------|
| `--backend` | config-only | config-only, unsloth, or axolotl |
| `--base-model` | unsloth/gemma-3-1b-it | HuggingFace model ID |
| `--lora-rank` | 16 | LoRA rank |
| `--epochs` | 3 | Training epochs |
| `--learning-rate` | 2e-4 | Learning rate |

`config-only` writes training scripts without running them. `unsloth` and `axolotl` also execute the training pipeline (requires Python environment). Requires exported training data from `export-training`.

#### validate-templates

Validate template JSON files for quality and correctness.

```
erenshor-llm.exe --data-dir data validate-templates [--check-grounding]
```

Checks format, forbidden phrases, duplicate IDs, category balance, and entity grounding against `grounding.json`. Available in development builds.

## SillyTavern Integration (TEST FEATURE)

The sidecar exposes a REST API that can be connected to [SillyTavern](https://github.com/SillyTavern/SillyTavern) for testing dialog outside the game. This lets you iterate on personality files, prompt tuning, and template quality without launching Erenshor.

### Setup

1. Start the sidecar manually:
   ```
   erenshor-llm.exe --data-dir data
   ```

2. In SillyTavern, configure a **Custom API** connection:
   - **API URL**: `http://127.0.0.1:11435`
   - **Endpoint**: `/v1/respond` (for dialog) or `/v1/paraphrase` (for event text)

3. Send requests matching the sidecar's JSON schema:
   ```json
   {
     "message": "Tell me about the Bone Pits",
     "sim_name": "Vesta",
     "zone": "The Bone Pits",
     "channel": "say",
     "player_name": "Hero",
     "relationship": 7.0
   }
   ```

### What You Can Test

- **Personality voice** -- Send the same message to different sim names and compare how Scrubby vs Evelia vs Clawsout respond. Verify archetype, tone, and vocabulary match the personality JSON.
- **Template coverage** -- Probe topics (lore, gear, zones, combat) to check if relevant templates surface. Gaps mean you need more templates.
- **LLM quality** -- Switch between `mode = "local"` and `mode = "cloud"` in the TOML to compare model outputs. Adjust temperature and max_tokens.
- **Lore grounding** -- Ask about deities, zones, items. Check that responses reference real Erenshor entities (GEPA validation catches hallucinations).
- **Paraphrase pipeline** -- Test event paraphrasing by sending canned game text through `/v1/paraphrase` with different triggers (group_death, loot_request, combat_callout, zone_entry).

### Offline Testing

The sidecar runs independently from the game. You can test the full intelligence pipeline -- embedding, template search, re-ranking, lore retrieval, LLM generation, GEPA validation -- without Erenshor running. This is useful for:

- Developing and validating new personality files before deploying
- Benchmarking model performance (see `tests/integration/bench-cloud-models.sh`)
- Bulk-testing template quality across many sim/zone/topic combinations
- CI/CD validation of template and lore changes

## Game Event Commentary

SimPlayers now react to in-game events with personality-driven commentary. Instead of canned messages like "Oh no!" when a group member dies, each SimPlayer processes the event through the full LLM pipeline to produce responses colored by their personality, knowledge, and relationship to the players involved.

### Supported Events

| Event | Trigger | Example |
|-------|---------|---------|
| Group member death | `group_death` | Scrubby: "Okay so Phanty died. Again. But hey, I almost died too, so at least we're consistent!" |
| Loot drops | `loot_request` | Rhys: "Eon Blade of Time? That's BiS for Windblades. I'd fight you for it." |
| Group invites | `group_invite` | Vesta: "Oh wonderful, more friends! The more the merrier, dear." |
| Combat callouts | `combat_callout` | Delivered instantly (skip LLM queue) -- "Pulling!", "Need heal!", "Incoming!" |
| Zone entry | `zone_entry` | Evelia: "The Bone Pits... the corruption here runs deep. Watch for Blight manifestations." |
| Level-up | `level_up` | Blademann: "Grats on 25. Now the REAL grind starts." |
| Hail/greeting | `hail` | Character-appropriate greeting based on personality and relationship |

### Priority Queue

Event paraphrasing uses a priority queue to prevent dialog flooding during combat:

- **Critical** -- Whispers to the player (always processed)
- **High** -- Death reactions, group invites (processed quickly)
- **Normal** -- Loot, zone entry (processed when queue has capacity)
- **Low** -- Ambient say/shout chat (processed when idle)
- **Skip** -- Combat callouts (delivered instantly as original text, no LLM delay)

Stale requests that have waited too long (configurable, default 4 seconds) are delivered as original text rather than blocking the dialog flow.

## Troubleshooting

**Mod loads but sims don't respond differently:**
- Check `BepInEx/LogOutput.log` for startup errors
- Verify the sidecar started: look for `erenshor-llm v0.3.0 starting on 127.0.0.1:11435`
- Make sure `.ruvector` files exist in `data/dist/`

**LLM responses are generic ("Hello."):**
- Ensure `LLM Mode` is set to `Local` (not `Off`) in BepInEx config
- Check that a GGUF model file exists in `data/models/`
- Look for shimmy startup messages in the log
- Try toggling `Restart Sidecar` to On

**Slow responses (>5s):**
- If on CPU, this is expected. GPU inference is 3-10x faster
- Try a smaller model (0.5B vs 1B)
- Reduce `Max Tokens` in BepInEx config
- Set `LLM Mode` to `Off` for instant template-only responses

**"Lore index not loaded" in health check:**
- Verify `data/dist/lore.ruvector` exists and is not 0 bytes
- Check that `erenshor-llm.toml` has correct index paths (should end in `.ruvector`)

**Sidecar crashes on startup:**
- Check if another process is using port 11435 (`netstat -an | findstr 11435`)
- Verify `onnxruntime.dll` is present alongside `erenshor-llm.exe`
- Try increasing `Startup Timeout` in BepInEx config

## Known Issues

- It's possible when quiting for sidecar erenshor-llm.exe, and shimmy.exe to hang making the game appear not to close.
- It's not perfect, you may see strange responses and interactions, especially if you use strange models.
- Template paraphrasing can occasionally produce slightly off-tone responses at very high temperatures (>1.0). Keep temperature at 0.7-0.8 for best results.
- The personality vector database rebuild at startup adds ~2-3 seconds to the first launch after updating personality files. Subsequent launches skip this if files haven't changed.
- Cloud/Hybrid LLM mode requires an OpenRouter API key configured in the sidecar TOML. The BepInEx config UI doesn't mask the key field.
- Memory accumulates over time. If `memory.ruvector` grows very large, you can delete it to reset conversation memory (sidecar creates a fresh one on next start).
